## 1. Project Overview

**NeighbourNet** is an offline mesh SOS communication system designed for disaster scenarios where cellular infrastructure fails (cyclones, floods, earthquakes). It turns every Android phone into a relay node in a self-forming Bluetooth/WiFi Direct mesh. SOS messages hop phone-to-phone without internet, are triaged by an on-device AI model, and are uploaded to a cloud dashboard the moment any device regains connectivity.

**Core value proposition:**
- Zero internet dependency for SOS send and relay
- On-device AI triage (CRITICAL / HIGH / MEDIUM / LOW) without a network call
- Automatic cloud sync + real-time coordinator dashboard when internet returns
- Works on any mid-range Android phone with no special hardware

**Target market:**
- NDRF / disaster relief coordinators (primary dashboard users)
- General population in flood/cyclone-prone regions (mobile app users)
- Emergency management NGOs

**Current stage:** Hackathon prototype (Hackathon 2026). Demonstrates the full end-to-end story. Not production-ready — has hardcoded identities, no auth, no multi-tenancy, and several stubbed features.

**Data flow — message creation to dashboard:**
1. Victim taps SOS → `SosScreen.tsx` calls `triageMessage(body)` (on-device)
2. Message created with UUID v4, GPS, priority_score/tier, TTL=10, hop_count=0
3. Stored in SQLite (`insertMessage`), `synced=0`
4. Broadcast via `NearbyModule.sendMessage()` (Kotlin → Nearby Connections API)
5. Relay phones receive via `onPayloadReceived` → dedup check → re-broadcast to all connected peers
6. When internet available: `startGatewaySync` fires `POST /api/messages/batch`
7. Backend deduplicates, triggers Gemini triage in background, updates Supabase
8. Supabase Realtime pushes UPDATE to dashboard WebSocket subscription
9. Dashboard shows ranked message, coordinator clicks Acknowledge

## 5. Database Schema

### 5.1 Supabase PostgreSQL — `messages` table

**File:** `NeighbourNetAPI/supabase/migrations/001_initial_schema.sql`

```sql
CREATE TABLE messages (
  -- Identity
  message_id          UUID            PRIMARY KEY,          -- UUID v4 from device
  sender_id           TEXT            NOT NULL,             -- Android ANDROID_ID

  -- Content
  body                TEXT            NOT NULL CHECK(char_length <= 500),
  body_hash           TEXT            UNIQUE,               -- SHA-256 dedup key

  -- Location
  gps_lat             DOUBLE PRECISION,
  gps_lng             DOUBLE PRECISION,
  location_hint       TEXT,

  -- On-device triage
  priority_score      FLOAT           NOT NULL DEFAULT 0 CHECK(0–1),
  priority_tier       TEXT            NOT NULL DEFAULT 'LOW' CHECK(CRITICAL|HIGH|MEDIUM|LOW),

  -- Cloud triage (Gemini, null until worker runs)
  cloud_priority_tier TEXT            CHECK(CRITICAL|HIGH|MEDIUM|LOW|NULL),
  triage_summary      TEXT,
  extracted_location  TEXT,

  -- Mesh metadata
  ttl                 INT             NOT NULL DEFAULT 10 CHECK(0–10),
  hop_count           INT             NOT NULL DEFAULT 0 CHECK(>=0),

  -- Timestamps (device clock may have skew — sort by last_hop_at, not created_at)
  created_at          TIMESTAMPTZ     NOT NULL,
  last_hop_at         TIMESTAMPTZ     NOT NULL,
  ingested_at         TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

  -- Sync state (always TRUE in the backend)
  synced              BOOLEAN         NOT NULL DEFAULT TRUE,

  -- Coordinator actions
  acknowledged        BOOLEAN         NOT NULL DEFAULT FALSE,
  acknowledged_at     TIMESTAMPTZ
);
```

**Indexes:**
- `idx_messages_priority_queue` on `(priority_score DESC, last_hop_at DESC)` WHERE `acknowledged = FALSE`
- `idx_messages_active_nodes` on `(sender_id, last_hop_at DESC)`
- `idx_messages_acknowledged` on `(acknowledged, priority_tier)`
- `idx_messages_tier` on `(priority_tier, last_hop_at DESC)` WHERE `acknowledged = FALSE`

**Realtime:** Enabled by running `ALTER PUBLICATION supabase_realtime ADD TABLE messages` in Supabase SQL editor (documented in migration comments, not automated).

**Missing table:** `mesh_telemetry` — referenced in `src/mesh/telemetryReporter.ts` but absent from the migration file. The telemetry reporter will fail silently.

## 7. On-Device AI Module

**File:** `NeighbourNet/src/services/triageEngine.ts`

### ONNX Runtime Integration
- **Model file:** `NeighbourNet/assets/model_int8.onnx` — **EXISTS BUT IS 0 BYTES (empty placeholder)**
- **Native module:** The TypeScript code references `NativeModules.OnnxTriage` with interface `{ classifyMessage(text): Promise<{priority_tier, priority_score, embedding?}> }`. **No Kotlin class implementing this interface exists in the codebase.** The only Kotlin module registered is `NearbyModule`.
- **Runtime behavior:** `OnnxTriage` is `undefined` at runtime. The `try/catch` in `triageEngine.ts:178–197` catches the undefined call and falls through to keyword fallback immediately.
- **Conclusion: ONNX triage is 100% stubbed. The keyword fallback is always active.**

### Keyword Fallback Scorer (Always Active)
Located in `triageEngine.ts`. Fully implemented:
1. `tokenize(text)` — splits on punctuation, takes first 128 tokens, converts to char-code sums
2. `textToSimpleVector(text)` — builds a 384-dimension vector, boosted by keyword hits
3. Critical keywords (Bengali + English): `trapped, rescue, emergency, unconscious, critical, আটকে, জরুরি, সাহায্য, উদ্ধার, নৌকা, বিপদ`
4. High keywords: `stranded, food, water, family, children, rising, খাবার, জল, পরিবার, বাড়ছে`
5. Low keywords: `safe, okay, fine, checking, ঠিক, আছি`
6. Priority score formula: `min(1, max(0, critical_sim + 0.7 × high_sim))`
7. Tier thresholds: CRITICAL ≥ 0.85, HIGH ≥ 0.65, MEDIUM ≥ 0.40

### Cloud Triage (Gemini)
Located in `NeighbourNetAPI/app/workers/gemini.py`.
- Uses `gemini-2.5-flash` (note: README and PRD say 1.5 Flash — actual code uses 2.5)
- Exponential backoff: 3 retries (1s / 2s / 4s)
- Demo cache: pre-populates result for `DEMO_MSG_ID` UUID to survive rate limits
- Prompt: multilingual (Bengali, English, Banglish), structured JSON output
- Markdown code fence stripping in response parser

## 8. Mesh Transport Layer

**Kotlin implementation:** `NeighbourNet/android/app/src/main/java/com/anonymous/NeighbourNet/NearbyModule.kt`  
**JS wrapper:** `NeighbourNet/src/services/meshService.ts`

### Peer Discovery
- **Status: ✅ Working**
- Strategy: `Strategy.P2P_CLUSTER` (fully decentralised, every node advertises + discovers)
- Service ID: app package name (`com.anonymous.NeighbourNet`)
- Connection initiation: device with lexicographically smaller endpoint name initiates (prevents race)
- `scanNow()` restarts discovery (stops then starts advertising + discovery)

### Message Forwarding Protocol
- **Status: ✅ Working (with caveats)**
- Payload: UTF-8 JSON bytes over Nearby Connections
- Dedup: bounded `LinkedHashSet<String>(500)` — evicts oldest on overflow
- Relay: `rebroadcastToAll(messageJson, excludeEndpointId)` — sends to all connected except source
- Directed vs broadcast: Both `destination_id=null` and `destination_id=set` use the same rebroadcastToAll. JS layer filters by UUID.

### TTL and Hop Count
- **Status: 🔨 Partial — TTL not decremented, hop_count not incremented during relay**
- The native `rebroadcastToAll` has `if (ttl <= 0) return` but `ttl` is a static class constant (`DEFAULT_TTL = 3`), not read from message JSON. This is effectively a global relay-enable flag.
- The message JSON's `ttl` field is never decremented when relayed.
- The message JSON's `hop_count` field is never incremented when relayed.
- `meshRelay.ts` checks `hop_count >= message.ttl` before re-queuing pending forwards, but since hop_count is never incremented, this check may not work as intended.

### Delivery Acknowledgements
- **Status: ❌ Dead code**

### Bugs and Incomplete Logic

1. **`model_int8.onnx` is empty (0 bytes)**  
   `NeighbourNet/assets/model_int8.onnx` — The ONNX triage model is a placeholder. OnnxTriage native module also doesn't exist. On-device AI is 100% keyword-based at runtime.

2. **`mesh_telemetry` Supabase table missing from migrations**  
   `src/mesh/telemetryReporter.ts` calls `supabase.from('mesh_telemetry').upsert(...)` but this table is not in `001_initial_schema.sql`. Every telemetry push will fail with a Supabase 404 and log a warning. The mesh visualisation page likely shows no real data.

3. **Hardcoded `sender_id = 'neighbournet-demo-user'`**  
   `src/screens/SosScreen.tsx:31` — Every SOS from every device has the same sender_id. The backend will count all devices as one node. The "active nodes" Prometheus gauge will never exceed 1 in production use with the demo app build.

4. **Hardcoded `gateway_id = 'neighbournet-mobile-gateway'`**  
   `src/services/gatewaySync.ts:17` — All gateways are indistinguishable in logs and metrics.

5. **Active nodes metric bug in `app/routes/messages.py:254–268`**  
   The query filters `last_hop_at >= now.replace(second=0, microsecond=0)` — this uses the current minute's start, not 30 minutes ago. The active nodes gauge reflects only messages from the current minute, not the last 30 minutes as intended. The `_refresh_gauges_loop` in `main.py` also uses a separate 30-minute calculation. The per-request update is effectively broken.

6. **TTL never decremented during relay**  
   Neither `NearbyModule.kt` nor `meshService.ts` nor `meshRelay.ts` decrement `ttl` in the message JSON when forwarding. The `hop_count` is also never incremented. Messages will relay indefinitely until the native dedup set catches them (which it always will after the first relay, since the same `message_id` is used).  
   Wait — the native dedup set catches duplicates correctly because the same `message_id` is seen again on the next hop. So in practice, a message travels through the mesh fine (each device relays it once and deduplicates future copies). The TTL/hop_count fields in the JSON are just metadata for analytics; they don't control relay in the current implementation.

7. **`sendAcknowledgement` dead code**  
   `NearbyModule.kt:137` — Never called. The acknowledgement delivery feature is not functional.

8. **Directed messages still broadcast to all**  
   `NearbyModule.kt:114–116` — Both `destination_id=null` and `destination_id=set` paths call `rebroadcastToAll`. Chat messages intended for a specific peer are still broadcast to everyone in the mesh (privacy concern, not a functional bug since JS layer filters by UUID).

9. **CORS wildcard origin**  
   `NeighbourNetAPI/app/main.py` — `allow_origins` includes `"*"`. Any website can make cross-origin requests to the API.

10. **`Supabase Realtime` not auto-enabled**  
    Migration SQL contains a comment saying to run `ALTER PUBLICATION supabase_realtime ADD TABLE messages` manually. If not done, dashboard Realtime subscriptions receive nothing.

## 10. SaaS Readiness Gaps

### 🔴 Blockers (cannot ship without these)

1. **No authentication anywhere** — API, dashboard, and all endpoints are fully open. Any actor can POST fake SOS messages, read all messages, or acknowledge them.
2. **Hardcoded demo identities** — `sender_id = 'neighbournet-demo-user'` and `gateway_id = 'neighbournet-mobile-gateway'` in production builds break multi-device tracking and analytics.
3. **ONNX triage is entirely unimplemented** — The on-device AI (primary differentiator) is 100% keyword scoring. The Kotlin native module and model file both need to be built. This is the RunAnywhere/Smolify prize feature.
4. **`mesh_telemetry` table missing** — Telemetry reporter fails on every push. Mesh visualization on the dashboard likely shows no real data.
5. **No foreground service** — The app will be killed by Android when the screen turns off, breaking mesh relay in real disaster scenarios.
6. **CORS wildcard** — Must be restricted to specific origins before production.
7. **No CI/CD pipeline** — No automated testing or deployment.

### 🟡 Important (ship soon after launch)

1. **No multi-tenancy** — Single Supabase table shared by all. An NGO's data is visible to any other user of the API.
2. **No rate limiting** — `/api/messages/batch` can be flooded; `/metrics` can be abused.
3. **Active nodes metric bug** — The per-request active nodes update is broken (calculates last minute, not last 30 minutes).
4. **TTL/hop_count not updated during relay** — Mesh analytics data is inaccurate.
5. **No boot receiver** — App doesn't restart after device reboot.
6. **`usesCleartextTraffic="true"`** — Must be removed; HTTPS required for production.
7. **Adaptive scan backoff** — Without it, continuous BLE/WiFi scanning will drain battery in 4–6 hours.
8. **No error reporting/alerting** — No Sentry, no uptime monitoring.
9. **No database connection pooling** — supabase-py creates a new client per process; fine for low load but needs review for production.
10. **`sendAcknowledgement` dead code** — Delivery confirmation feature is non-functional.

### 🟢 Nice to Have

1. iOS support (PRD documents this as out of scope for v1)
2. End-to-end encryption / message signing
3. Two-way coordinator-to-victim messaging
4. Offline OSM tile caching
5. LoRa/Meshtastic bridge for long-range extension
6. Dynamic TTL based on network density
7. BLE-only low-power mode toggle
8. Relay runner protocol UI (for mesh partition scenarios)
9. Coordinator audit log (who acknowledged what, when)
10. API versioning (`/v1/api/messages`)
